# 발표 기획: AP 설계팀 Hardware Debugging Process 자동화

## 발표 개요

| 항목 | 내용 |
|------|------|
| 세션 주제 | AI/자동화 활용 사례 공유 |
| 발표 제목 | AP 설계팀의 Hardware Debugging Process 자동화 |
| 발표 시간 | 1시간 |
| 청중 | 개발+비개발 혼합 (HW 도메인 기본 지식 있음) |
| 발표 형식 | 슬라이드 + Web UI 라이브 데모 |

---

## 핵심 메시지

> "자동화에 꼭 AI/ML이 필요하지 않다. 기존 데이터 자산 + 도메인 이해 + 단순한 도구 = 전문가의 반복 작업 제거"

**발표의 가치**: 기술 복잡도가 아니라 **문제 발굴 능력 + 데이터 자산화 인사이트**

**청중이 가져가야 할 것**: "나도 우리 팀에서 이런 기회를 찾을 수 있겠다"

---

## 1시간 발표 구성

### 1부. 문제: HW 디버깅 병목 (10분)

**목표**: 청중이 "이건 진짜 힘들겠다"를 느끼게 하기

- 스토리텔링으로 시작
  - "칩이 처음 나왔다. 6개월, 수백억 원. 그런데 버그가 있다."
- 디버깅 프로세스 흐름 설명 (Scandump → Regbank 생성 → 분석)
- 비포: 각 담당자가 자체 스크립트를 보유하고 있지만, 이슈가 담당자에게 할당되어야 비로소 분석이 시작됨
  - 담당자를 찾아나서는 과정 자체가 병목
  - 내 시스템: 담당자를 거치지 않고 Web UI에서 누구나 바로 확인 가능

---

### 2부. 사전 준비: 데이터 자산화 (10분)

**목표**: 자동화의 전제조건이 "데이터 자산화"였음을 강조

- 이번 시스템이 가능했던 이유: 설계팀 Spec 데이터를 미리 DB로 자산화해두었기 때문
  - 당시에는 당장 활용처가 불명확했지만, 이 자산이 없었다면 자동화는 불가능했다

- 핵심 인사이트: **AI Agent/자동화를 하고 싶다면, 먼저 데이터를 자산화해야 한다**
  - 데이터가 없으면 LLM도, 파싱도, 어떤 자동화도 할 수 없다
  - 지금 쓸모없어 보이는 데이터 수집이 미래의 자동화 가능성을 열어준다

- 연결: HW 팀은 Regbank를 항상 생성하고 있었다 → **두 자산을 연결**하는 것이 이번 시스템의 핵심

---

### 3부. 솔루션: 무엇을 만들었나 (15분)

**목표**: 시스템 이해 + 라이브 데모로 임팩트 전달

- 아키텍처 다이어그램 (1장)
  ```
  ┌─────────────────┐   ┌─────────────────────┐
  │   Spec Data DB  │   │  Regbank Storage    │
  │  (설계 Spec 자산) │   │ (이슈별 Regbank 보관) │
  └────────┬────────┘   └──────────┬──────────┘
           │                       │
           └──────────┬────────────┘
                      ▼
             ┌─────────────────┐
             │  파싱 / 분석 엔진  │
             └────────┬────────┘
                      ▼
             ┌─────────────────┐
             │     Web UI      │
             │   (결과 시각화)   │
             └─────────────────┘
  ```

- **라이브 데모** (약 10분)
  1. Regbank 파일 업로드
  2. 처리 과정 간략 시연
  3. Web UI에서 분석 결과 확인
  4. "엔지니어가 수동으로 했다면 이 결과를 얻는 데 상당한 시간이 필요했을 것"

- 기술 세부사항은 최소화 → **결과물의 가치**에 집중

---

### 4부. 여정: 어떻게 만들었나 (10분)

**목표**: SW↔HW 협업 과정 공유 + 현재 한계와 가능성 제시

- **SW 담당자의 시각**: HW 디버깅 도메인 지식 없이 시작했다
- **협업이 핵심**: 도메인 문제 정의는 HW 엔지니어, 기술 구현은 SW 담당자
- 예상보다 어려웠던 것 (솔직하게)
  - 디버깅 프로세스 도메인 이해
  - Spec 데이터 정제 및 구조화
  - 엣지 케이스 처리
- 기술적 단순함은 의도적 선택: 복잡하지 않게 만든 것이 강점
- **현재 한계 + 진화 가능성**
  - 현재: 파싱/분석 자동화 수준
  - 디버깅 프로세스를 더 깊이 이해할수록 더 진화된 Agent 개발 가능
  - 도메인 지식이 쌓이면 문제 원인 추론까지 자동화 가능성 있음

---

### 5부. 프레임워크: 당신 팀에서도 찾을 수 있다 (10분)

**목표**: 청중이 자기 팀에 적용할 수 있는 사고방식 전달

**자동화 기회 발굴 3가지 질문**:
1. 우리 팀에 반복적으로 수행하는 전문가 작업이 있는가?
2. 그 작업에 사용되는 데이터가 이미 어딘가에 구조화되어 있는가?
3. 두 데이터를 연결하면 자동화할 수 있는가?

> 청중에게 5분 정도 자기 팀 케이스를 생각해보게 유도하면 참여도 높아짐

---

### 6부. 마무리 + Q&A (5분)

- 요약 슬라이드 1장: 문제 → 자산화 → 솔루션 → 프레임워크
- 현재 상태 솔직하게: "아직 초기 단계, 측정 데이터는 다음 발표에서 공유하겠다"
- **클로징 멘트**:
  > "자동화와 Agent를 더 발전시키고 싶은데, 어떤 Framework로 설계하면 좋을지 아직 고민 중입니다. 같은 고민을 하고 계신 분이 있다면 발표 후 함께 이야기 나눠보고 싶습니다."

## 정량 데이터 없는 문제 대응

초기 단계라 수치가 없으므로:
- "현재 파일럿 중, 초기 피드백은 긍정적" 방식으로 전달
- 예상 효과 기반: "디버그 세션 1회당 수동 분석 시간을 [추정치]만큼 줄일 수 있을 것으로 기대"
- 솔직하게: "아직 측정 데이터는 없지만, 다음 발표에서 공유하겠다"
  - 솔직함 = 신뢰 + 다음 발표 예고 효과
